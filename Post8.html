Attribution and Deep Link Support Lab

Simulating Real SaaS Tracking Issues, API Debugging, and Customer-Facing Investigations

Posted by John Plakon on March 9, 2026

Introduction

This project was built to simulate the kinds of technical issues commonly encountered in mobile SaaS, mar-tech, and attribution platforms. The lab combines a lightweight event tracking API, a mock campaign link flow, SQL-based investigation queries, and support runbooks that document how issues were reproduced, diagnosed, and resolved.

Instead of focusing only on analytics outputs, this project emphasizes the investigative process behind technical support: inspecting payloads, validating redirects, tracing event loss, comparing raw logs against reporting tables, and translating technical findings into clear guidance for customers and internal teams.

The goal was to create a realistic support engineering portfolio piece that demonstrates practical troubleshooting skills with APIs, SQL, browser tools, event pipelines, and customer communication.

Project Goals

The lab was designed around four goals:

  • Simulate attribution and deep linking workflows across click, open, install, and conversion events.
  • Reproduce realistic data and tracking failures that a support engineer would investigate.
  • Use SQL, logs, and browser network inspection to identify root causes.
  • Document findings in a format suitable for customer support, knowledge base articles, and engineering handoff.

System Overview

The project includes a simple web landing page, a mock redirect flow, an API service for recording event traffic, and a small event warehouse for investigation. Clicks, opens, installs, and conversions are captured with campaign metadata and device identifiers, allowing end-to-end validation of attribution behavior.

By intentionally introducing failure cases into the flow, I was able to simulate support scenarios such as missing conversions, broken deep links, campaign mismatches, duplicate events, malformed payloads, and reporting discrepancies between raw data and final dashboards.

Core Components

  • Mock tracking API: receives click, install, open, and conversion events.
  • Redirect and deep link flow: simulates campaign links, fallback pages, and app-opening logic.
  • SQLite/Postgres event store: stores raw event data for root cause analysis.
  • SQL investigation queries: validates attribution chains and identifies broken event paths.
  • Support runbooks: explains how each issue was reproduced and resolved.
  • Incident documentation: provides postmortem-style writeups for recurring issues.

Example Event Flow

A normal event path in the lab follows this sequence: 

1. User clicks a campaign link
2. Redirect service stores click metadata
3. User opens the app or fallback page
4. Install/open event is recorded
5. Conversion event is submitted
6. SQL logic attributes the conversion back to the original campaign

This controlled flow makes it easy to compare expected behavior against real event output when something fails.

Support Scenarios Simulated

I created multiple realistic support cases to reflect the types of technical problems that arise in attribution and growth platforms:

  • UTM parameters dropped during redirect.
  • Conversion received without a matching click.
  • Duplicate conversion firing due to client retry behavior.
  • Deep link opening fallback web content instead of target app content.
  • Campaign source mismatch between raw event payload and final reporting table.
  • Timezone mismatch causing reporting delays and confusion.
  • Malformed payload accepted by endpoint but excluded from attribution logic.
  • Cross-domain/browser behavior breaking session continuity.

Sample Investigation: Missing Conversion Attribution

One of the most useful scenarios in the project involved a click event successfully entering the system while the downstream conversion failed to attribute to the originating campaign.

I approached the issue by validating the raw click event, confirming whether the install and open event carried the same identifier, and then checking whether the conversion payload preserved the expected campaign key. The investigation showed that the conversion request was missing a required field after a redirect step stripped campaign context from the session. 

SELECT
  c.click_id,
  c.campaign_name,
  c.device_id,
  c.click_timestamp,
  v.conversion_timestamp,
  v.revenue
FROM click_events c
LEFT JOIN conversion_events v
  ON c.device_id = v.device_id
WHERE v.conversion_timestamp IS NULL;

This query made it easy to identify clicks that never produced an attributed downstream conversion. From there, I compared payload structures and redirect behavior to isolate the missing handoff field.

Sample Investigation: Duplicate Events

Another scenario involved duplicate conversion events entering the database because the client retried requests without idempotency protection. The symptom from a support perspective was inflated conversion totals in reporting and inconsistent campaign ROI.

After reviewing event timestamps and payload values, I created a query to detect repeated conversion submissions for the same user, event name, and campaign within a short interval. 

SELECT
  device_id,
  campaign_name,
  event_name,
  COUNT(*) AS duplicate_count
FROM conversion_events
GROUP BY device_id, campaign_name, event_name
HAVING COUNT(*) > 1;

The root cause was not a reporting bug but a client-side retry pattern. I documented both the technical explanation and a customer-friendly explanation so that the issue could be escalated to engineering while still giving the customer a clear answer.

How I Investigated Issues

This project intentionally mirrors the workflow I would use in a technical support engineering role:

  • Replicate the issue in a controlled environment.
  • Inspect browser behavior using Chrome Developer Tools.
  • Validate API requests, response codes, and payload structure.
  • Query the warehouse to compare raw inputs against expected outputs.
  • Separate product defects from expected behavior or integration mistakes.
  • Write a support summary that explains impact, cause, and next steps.

Support Runbooks

A major focus of this project was building documentation, not just code. For each scenario, I wrote an internal runbook that included:

  • Customer symptom
  • Reproduction steps
  • Logs or payload examples
  • SQL checks
  • Likely root causes
  • Resolution path
  • Suggested customer response

This reflects how support engineers scale knowledge across teams and improve response quality over time.

Customer-Facing Communication

I also included mock ticket responses to demonstrate how technical findings can be translated into clear, actionable updates for customers. In support roles, solving the issue is only part of the job; the other part is communicating the diagnosis in a way that is accurate, concise, and easy to follow.

The project therefore includes both technical notes for internal teams and customer-facing summaries that avoid unnecessary jargon while still setting correct expectations.

  • Cross-functional thinking between support, product, engineering, and analytics

Why This Project Matters

Attribution and event tracking issues are often difficult because the failure does not always occur where the customer first notices it. A dashboard discrepancy may actually be caused by a redirect problem, a malformed payload, a browser constraint, duplicate retries, or a downstream logic rule. This project demonstrates my ability to investigate those issues systematically rather than relying on surface-level symptoms.

It also reflects the kind of work performed in high-volume SaaS support environments where accuracy, clarity, and speed all matter.

Conclusion

This support lab was built to bridge analytics, application troubleshooting, and customer-facing technical investigation. It highlights not just my ability to work with data, but my ability to diagnose end-to-end technical issues, explain root causes, and document repeatable solutions.

For teams supporting APIs, SDKs, deep links, and attribution pipelines, this kind of project demonstrates the practical mindset required to handle real customer issues and collaborate effectively across support, product, and engineering.
